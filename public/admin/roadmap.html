PENDÊNCIA

Afiliados de remessa com link real

Hoje em produção: AFFILIATE_WISE_LINK, REMITLY, WU e KAYAK têm env vars setadas mas o conteúdo é literal https://wise.com/invite/placeholder — alguém colou placeholder e nunca trocou.

How to apply: cadastrar nos programas (Wise Partnerize, Remitly Impact, etc.), pegar o link real e atualizar na Vercel → Settings → Environment Variables. Detalhes em AUDITORIA_PARCERIAS.md.

GA4 + Meta Pixel

Tracking de eventos e retargeting estão prontos pra plugar. Faltam só os IDs.

How to apply: criar property em analytics.google.com e Pixel em business.facebook.com, passar os IDs e a gente seta as env vars VITE_GA4_MEASUREMENT_ID e VITE_META_PIXEL_ID.

Push notifications — VAPID

✅ Já configurado em produção (VAPID_PUBLIC_KEY, VAPID_PRIVATE_KEY, VAPID_SUBJECT setados). /api/push/vapid retorna configured: true. PushPrompt e opt-in contextual funcionando.

Magic link customizado (Supabase + Resend)

Hoje o magic link sai do remetente padrão Supabase com template em inglês. Pra trocar por algo branded de oi@brasilconnectusa.com, faz 3 coisas no dashboard.

How to apply: seguir docs/supabase-auth-setup.md (Custom SMTP via Resend + colar template docs/email-magic-link.html + whitelistar Redirect URLs).

Migration bc_classified_status

Marketplace MVP precisa da coluna nova em bc_posts. Sem isso, criar post tipo classified vai falhar.

How to apply: rodar supabase/bc_classified_status.sql no SQL Editor do Supabase Dashboard antes de mergear o PR do marketplace.

Resend domain verification

Sem brasilconnectusa.com verificado no Resend (SPF + DKIM verde), os emails de boas-vindas + magic link caem em spam ou não saem.

How to apply: resend.com/domains → adicionar domínio + colar registros DNS.

Próximo (pós-lançamento do Bolão)

Ponte entre estática e app

Hoje quem entra em /guias/cnh/ não vê o app. Quem está em /app/feed não vê os guias. Cada lado fica isolado.

  • Adicionar header consistente com link "App" nas páginas estáticas
  • Adicionar item "Guias" no menu do app linkando pra /guias/
  • Considerar uma "topbar global" idêntica em ambos os mundos

OG image dinâmico (PNG)

Hoje usamos SVG estático. Alguns scrapers (FB/IG mais antigos) preferem PNG. Soluções: converter manualmente o SVG existente pra PNG, ou adicionar @vercel/og pra gerar imagem dinâmica per-grupo (com nome do grupo, contador de membros, etc).

Tracking de eventos no app

Quando GA4 + Pixel estiverem ativos, instrumentar eventos chave: bolao_grupo_criado, bolao_palpitou, bolao_convite_enviado, remessa_clique, voos_buscou. Permite medir conversão real do funil.

Push: cobrança de palpites

Cron diário que dispara push pra membros que ainda não palpitaram, lembrando do prazo. Tópico bolao já existe e usuário já pode optar in via banner pós-criação de grupo.

Sub-guias específicos de imigração

A página atual é overview. Quando tiver volume e validação jurídica, expandir cada caminho em sub-páginas estilo /guias/cnh/: Green Card detalhado (4 caminhos), H-1B, L-1, EB-2 NIW, asilo passo-a-passo, N-400 (naturalização), wage theft (DOL).

Why: trabalho é ~60% conteúdo (pesquisa + revisão jurídica) e 40% código. Página overview já cobre o essencial pra captar tráfego SEO.

🎟️ Eventos com ticketing pago + take rate da plataforma

O tab "Eventos" no AppShell hoje é placeholder. Decidimos NÃO lançar versão grátis sozinha — fazer direto com monetização integrada. Modelo: organizador cria evento → vende ingresso via BrasilConnect → BC cobra % do valor (estilo Eventbrite/Sympla mas focado em comunidade BR-USA).

Componentes da feature completa:

  • Schema bc_events + bc_event_tickets + bc_event_orders (com Stripe Connect pro split organizador/plataforma)
  • Eventos grátis (RSVP simples) e pagos (Stripe Checkout)
  • Organizador onboard via Stripe Connect Express (igual já fazemos pra restaurantes em api/restaurant/onboard.js)
  • Take rate: 5-10% do valor do ingresso (calibrar vs Eventbrite que cobra 3.5% + $1.79/ticket). Definir em application_fee_amount via Stripe Connect
  • Lista de inscritos pro organizador (export CSV, check-in via QR code no dia)
  • Email de confirmação com QR code do ticket
  • Reembolso via UI (organizador aprova, Stripe processa)
  • Política anti-spam: só negócios verificados podem criar eventos pagos (evita golpe)

Trabalho prévio que ficou no working tree (não mergeado): branch claude/events-feature (PR #31 fechada) tem um MVP free de eventos (schema + API list/submit + tela). Vale referência mas precisa rework pra incluir ticketing.

Why: Eventos é catalisador de comunidade (festas BR, igrejas, encontros profissionais). Sem monetização vira só "feed de eventos" — pouco diferenciado vs Facebook Events. COM ticketing + 5-10% take rate, vira fonte de receita recorrente alinhada com o pacote "Restaurant Connect" que já temos (Stripe Connect reutilizável).
How to apply: Esperar lançamento do Bolão Copa estabilizar (D+30). Daí design session com Anderson: definir take rate exato, fluxo de organizador, política de aprovação. Stack reutiliza Stripe Connect já implementado. ETA dev: 7-10 dias.

Estratégico (decisão grande, depois da Copa)

LATER

Migrar pra Next.js (ou Astro) — em 3 fases

Hoje a stack é 3 paradigmas: Vite SPA (/app/*), HTML estático (landing, /guias, /negocio) e serverless functions (api/*). Funciona, mas duplica conteúdo entre estática e React (o caso mais visível: /negocio e /app/negocio mostram coisas diferentes).

Validação externa: Nextdoor passou pelo mesmo dilema e migrou pra Next.js exatamente pra resolver SEO de páginas de negócios + manter SPA pra experiência logada. SSR + SPA híbrido é a arquitetura final pra esse tipo de produto.

Fases propostas:

  • D+15 (logo após launch): plugin SSG no Vite (vite-plugin-ssr ou react-snap) — gera HTML estático do React no build com dados do Supabase. Resolve 80% do SEO sem trocar framework. ~2-3 dias.
  • D+60 (se SEO virar prioridade real): migra /negocio + /guias pra Astro (mantém Vite no resto). Astro foi feito pra esse caso. ~5-7 dias.
  • D+90+ (se virar produto multi-cliente ou precisar de middleware): migração full Next.js (App Router + Server Components). ~2-4 semanas.
Why considerar: conforme o catálogo de negócios e guias cresce, SEO orgânico vira fonte primária de tráfego frio. SPAs são quase invisíveis pro Google. Duplicar header/footer/auth entre estática e React também pesa na manutenção.
How to apply: NÃO fazer durante Copa 2026 nem nas primeiras 4 semanas pós-launch. Avaliar fase 1 (Vite-SSG) em julho/agosto quando o pico passar.

Auth federada (Google / Apple) no app

Hoje o app usa email + Supabase auth quando o user faz onboarding. Pra reduzir friction, considerar OAuth com Google e Apple — especialmente útil pro Feed e Bolão (onde a barreira de criar conta hoje é alta).

Dashboard de comunidade por cidade

Hoje as comunidades existem mas não têm página dedicada com ranking de membros, posts pinados, eventos próximos. Pode virar a "praça pública" de cada cidade brasileira nos EUA.

App nativo (iOS/Android via Capacitor ou Expo)

O PWA atual já permite "Add to Home Screen". Pra ter ícone na App Store + push iOS confiável, considerar wrapper nativo. Apenas se métricas mostrarem que o PWA tá perdendo retenção vs apps nativos concorrentes.
